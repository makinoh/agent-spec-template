## 用語定義（Definitions）

本書で繰り返し用いる用語を以下に定義します。

性質の異なる複数のAIツール（コーディング支援、自律実行型エージェント等）を一律に扱わないため、役割に着目して定義します。

| 用語                          | 定義                                                                  |
| --------------------------- | ------------------------------------------------------------------- |
| AIエージェント（Agent）            | コードまたはドキュメントを生成・編集するAIシステムの総称。                                    |
| 自律エージェント（Autonomous Agent） | 人間の逐次的な介入なしに、複数ステップのタスクを継続実行しうるAIエージェント。                          |
| 対話型エージェント（Interactive Agent） | 人間の指示・確認を逐次受けながら動作するAIエージェント。                                     |
| 人間承認者（Human Approver）       | 「6. AIエージェント統治と自律境界」の承認マトリクスにおいて承認権限を持つ人間。                        |
| 保護対象ブランチ（Protected Branch）  | ブランチ保護を有効化し、品質ゲートと必須レビューの通過を反映条件とするブランチ（例：main、リリースブランチ）。対象の具体は development-process.md で定義する。 |
| エージェント指示ファイル（Agent Instruction File） | AIエージェントが自動読込する実行指示文書。AGENTS.md を正本とし、CLAUDE.md 等のツール固有ファイルを含む。 |

本書で「AI」と総称する場合、特に断りのない限り上記のAIエージェント全般を指します。自律的な行為の可否は、ツールの種別ではなく本書の定める行為類型および変更クラスによって決まります。

## 1. RFC 2119 / RFC 8174 準拠

本書では RFC 2119 および RFC 8174 に従い、以下の用語を規範的な意味で使用します。

| 用語         | 意味    |
| ---------- | ----- |
| MUST       | 必須    |
| MUST NOT   | 禁止    |
| SHOULD     | 強く推奨  |
| SHOULD NOT | 原則非推奨 |
| MAY        | 任意    |

これらのキーワードは大文字で記載された場合のみ、RFCで定義された規範的意味を持ちます。

RFCキーワードを含まない記述（「〜とします」等）は、構成・事実の宣言または説明であり、規範的義務を課しません。遵守を求める規範には、必ず上記キーワードを明記します。ただし、明示的に「規範ではない」と宣言した派生サマリ（クイックリファレンス、改正履歴、索引等）に現れるキーワードは、本文規範の要約参照であり、当該サマリ自体には規範的効力を与えません。

---

### 1.1 義務レベルと強制手段（AI駆動開発における運用）

RFC 2119 のキーワードは *義務の強さ* を表します。

一方、そのルールを *どの手段で強制するか*（機械強制か人間ゲートか）は、義務の強さとは独立した軸です。

本プロジェクトでは両者を分離して扱います。

強制手段は次の3種類とします。

| 強制手段             | 内容                                                       |
| ---------------- | -------------------------------------------------------- |
| 機械強制             | CI/CD または自動検証により、反映前に強制する（「8. 機械的に検証可能なルール」に登録）          |
| 人間ゲート            | 必須レビュア・CODEOWNERS 等により、反映前に人間が承認する（「6. AIエージェント統治と自律境界」承認マトリクス） |
| 構造的強制            | 権限・環境の設計により、違反行為自体を不可能にする（例：本番接続権限の不付与、環境分離）。適用可能な場合は最優先とする |
| 未整備（ブートストラップ）    | 強制手段が未整備の期間。必須の人間レビューで暫定的に担保する（「8. 機械的に検証可能なルール」ブートストラップ規定） |

運用原則は以下のとおりです。

* 義務として課すべきルールは MUST / MUST NOT、推奨にとどめるルールは SHOULD / SHOULD NOT を用います（義務の強さでキーワードを選ぶ）。
* MUST / MUST NOT は可能な限り機械強制を優先します。  
  機械強制が困難でも義務として課す必要があるもの（例：本番データ操作の禁止、AIによる権限影響の開示）は、MUST / MUST NOT のまま人間ゲートで強制します。
* SHOULD / SHOULD NOT であっても、機械検証が容易なものは自動検証を併用してもよい（MAY）。  
  義務レベルと強制手段は独立に決まります。
* 機械強制ルールとして定義した MUST / MUST NOT（8章）は、CI/CD または自動検証によって実際に強制されていなければなりません（MUST）。
* 人間ゲートで担保する MUST / MUST NOT（6章 承認マトリクス）は、必須レビュア・CODEOWNERS 等によって反映前に強制されなければなりません（MUST）。
* 強制手段が未整備の MUST / MUST NOT を、整備済みであるかのように扱ってはなりません（MUST NOT）。詳細は「8. 機械的に検証可能なルール」のブートストラップ規定に従います。
* 本書のすべての MUST / MUST NOT に強制手段（整備前はブートストラップ）が対応することの網羅性は、「8. 機械的に検証可能なルール」の強制台帳規定が担保します。

## 2. 文書管理階層

ドキュメントの優先順位は以下の通りです。本章を文書階層の正本（SSoT）とし、他章はこの優先順位を参照します。

1. 開発憲章（constitution.md）
2. 憲章改正記録（governance/*）
3. プロジェクト基本方針（charter.md / vision.md / scope.md）
4. ADR運用規則（adr-rules.md）
5. アーキテクチャ決定記録（adr/*）
6. 開発プロセス（development-process.md）
7. アーキテクチャ設計（architecture/*）
8. 横断的技術標準（standards/* ※ ai-governance.md を除く）
9. AIガバナンス標準（standards/ai-governance.md）
10. エージェント実行指示（AGENTS.md。CLAUDE.md / GEMINI.md / SKILLS.md 等のツール固有ファイル・能力カタログを含む）
11. エージェント知識・手順（knowledge/ / playbooks/ / prompts/）
12. 機能概要説明（README.md）
13. ソースコードコメント

下位文書が上位文書と矛盾する場合、上位文書を優先しなければなりません（MUST）。

charter.md / vision.md / scope.md は、プロジェクトの目的・ビジョン・スコープ（「なぜ・何を」）を定める基本方針文書であり、設計判断（ADR）より上位に位置づけます。

adr-rules.md は ADR の運用規則（命名規則・必須セクション・Status 語彙等）を定めるメタ文書であり、本書に従属し、個別 ADR の構造的事項には優先します。

AIガバナンス標準（standards/ai-governance.md）は standards/ に属する一文書であり、AI運用に関する詳細方針を定めます。AI運用に関する規範は、本書（原則）→ AIガバナンス標準（詳細方針）→ エージェント実行指示（AGENTS.md および CLAUDE.md 等のツール固有ファイル）の三層で責務を分離します（6章参照）。

開発憲章そのものの改正は「ガバナンス決定」であり、アーキテクチャ決定であるADRとは種類が異なります。

両者は分離して管理するべきです（SHOULD）。

詳細は「7. 変更管理」を参照します。

## 3. 基本原則

### ドキュメントファースト

重要な知識を口頭・チャット・会話履歴のみに依存してはなりません（SHOULD NOT）。

再利用が必要な知識はドキュメント化し、リポジトリで管理するべきです（SHOULD）。

---

### ADRファースト

長期的な影響を持つ設計判断はADRとして記録するべきです（SHOULD）。

設計意図を失う実装先行は避けるべきです（SHOULD）。

判断理由は明文化するべきです（SHOULD）。

ADRの作成漏れは、それ自体が検知困難です。

本プロジェクトでは「ADRを書かなかったこと」を検知するため、重大な変更を含むPRにADR参照またはADR不要理由の記載を求めます（「5. ADRポリシー」参照）。

---

### 仕様ファースト（Specification First）

Class A または Class B に該当する実装を、対応する要求・仕様が存在しない状態で開始するべきではありません（SHOULD NOT）。

実装は仕様に追従するべきです（SHOULD）。

仕様と実装が乖離した場合、実装の修正または仕様の更新のいずれかで解消するべきです（SHOULD）。どちらで解消するかは人間が判断するべきです（SHOULD）。

仕様の変更を伴う実装変更は、対応する仕様更新を同一の変更単位（PR等）に含めるべきです（SHOULD）。

仕様の正本の所在・粒度・テンプレートは development-process.md で定義するべきです（SHOULD）。

仕様は「何を作るか（要求・振る舞い）」の正本であり、ADRは「なぜこの設計か（判断・根拠）」の正本です。両者は補完関係にあります。

---

### AI駆動開発（AIDD: AI-Driven Development）

本プロジェクトはAI駆動開発を前提とします。

AIエージェントは継続的な開発メンバーとして扱うべきです（SHOULD）。

AIが人間と同等に判断根拠へアクセスできる状態を維持するべきです（SHOULD）。

AIエージェントの自律的な行動範囲と人間の承認が必要な行為は、「6. AIエージェント統治と自律境界」で明示的に定義するべきです（SHOULD）。

---

### 信頼できる唯一の情報（SSoT: Single Source of Truth）

同一情報を複数箇所で管理してはなりません（SHOULD NOT）。

各情報は信頼できる唯一の情報源（Source of Truth）を持つべきです（SHOULD）。

| 情報            | 正本                          |
| ------------- | --------------------------- |
| プロジェクト目的      | charter.md                  |
| ビジョン          | vision.md                   |
| スコープ          | scope.md                    |
| 開発プロセス        | development-process.md      |
| 設計判断          | ADR                         |
| ADR運用規則       | adr-rules.md                |
| AIガバナンス詳細方針   | standards/ai-governance.md  |
| エージェント実行指示    | AGENTS.md（ツール固有差分は CLAUDE.md / GEMINI.md / CODEX.md / OPENHANDS.md / TAKT.md 等。名簿・協調は agents/） |
| エージェント能力（スキル） | SKILLS.md（→ skills/）       |
| マルチエージェント名簿・協調 | agents/（実行指示の正本は AGENTS.md） |
| エージェント作業記憶    | memory/（非規範。確定後は adr/・knowledge/ 等へ昇格） |
| ドメイン知識        | knowledge/                  |
| 再利用タスク手順      | playbooks/                  |
| プロンプト資産（ライフサイクル統治） | prompts/             |
| ドメイン用語（ユビキタス言語） | glossary.md              |

| 技術標準          | standards/                  |
| 計測（DORA・AI有効性） | metrics/                    |
| ガバナンス決定（改正提案・確定記録） | governance/decisions/（「7. 変更管理」参照） |
| ガバナンス例外・適用除外・リスク | governance/exceptions/・waivers/・risk-register/ |
| 文書優先順位        | 本書「2. 文書管理階層」               |

コードから生成可能な定義は、生成元を唯一の正本（SSoT）とし、生成物を直接編集してはなりません（SHOULD NOT）。

例として、API定義は OpenAPI、データモデルは Prisma 等のスキーマ、インフラ定義は Terraform 等を正本とします。

生成物が正本と一致していることは、CIで生成処理を再実行し差分が無いことを検証することで機械的に確認できます（SHOULD）。

これにより、レビュー依存になりがちなSSoT遵守の一部を自動検出可能にします。

重要な判断は理由と共に記録するべきです（SHOULD）。

### 原則の競合と裁定（Conflict Resolution）

文書間の競合は「2. 文書管理階層」に従います（本節では再掲しません）。

同一文書内の原則どうしが競合する場合、適用される法令・規制を最優先しなければなりません（MUST）。

適用される法令・規制・認証要件は、識別し文書化しなければなりません（MUST。正本は governance/ または standards/security-standards.md）。

安全側の原則（Security by Default、データ保護とAI入力境界、監査証跡）は、開発速度・利便性・コストの考慮に優先するべきです（SHOULD）。

AIエージェントは、競合を自らの判断で解消できない場合、自律判断せず人間の裁定を仰がなければなりません（MUST。「6. AIエージェント統治と自律境界」のHITLチェックポイント参照）。

裁定の結果は、ADRまたはガバナンス決定として記録するべきです（SHOULD）。

### Documentation as Code（DaC）

ドキュメントはコードと同等に扱うべきです（SHOULD）。

ドキュメントはレビュー対象とするべきです（SHOULD）。

ドキュメントはGitで管理し、変更履歴を追跡可能な状態を維持するべきです（SHOULD）。

---

### Security by Default

セキュリティ要件を後付けにしてはなりません（SHOULD NOT）。

認証、認可、秘密情報管理は設計段階で考慮するべきです（SHOULD）。

秘密情報（認証情報、APIキー、トークン等）をソースコードまたは設定ファイルにハードコードしてはなりません（MUST NOT）。

データは機密区分を定め、区分に応じた取り扱いをするべきです（SHOULD）。

機密区分の定義と取り扱い基準は standards/security-standards.md で管理するべきです（SHOULD）。

セキュリティ上の重要な設計判断はADRとして記録するべきです（SHOULD）。

---

### データ保護とAI入力境界

AIエージェントおよび外部AIサービスに対し、本番環境の個人データ、顧客機密、および秘密情報（認証情報・APIキー・トークン等）を入力してはなりません（MUST NOT）。

本ルールは構造的強制を第一とします。AIエージェントには、本番の個人データ・顧客機密に到達しうる接続権限を付与してはなりません（MUST NOT）。人間ゲート、シークレットスキャン、およびログ監査で補完します。

開発・検証には、匿名化済みデータ、マスク済みデータ、または合成データを使用するべきです（SHOULD）。

データ機密区分とAIツールの許容される組み合わせ、および外部AIサービスへの送信可否の判定基準は、standards/security-standards.md および standards/ai-governance.md で定義しなければなりません（MUST）。

個人データの取り扱い基準（収集の最小化・保存期間・利用目的の制限を含む）は、standards/security-standards.md で定義しなければなりません（MUST）。

個人データ・顧客機密を収集・記録（ログを含む）・送信しうる実装に着手する時点で、AIエージェントは機密区分とAIへの入力可否の基準（standards/security-standards.md・standards/ai-governance.md）に照合しなければなりません（MUST）。当該基準が未整備の場合は「未定義」として扱い、「8. 機械的に検証可能なルール」ブートストラップ規定に従い、人間の判断を仰がなければなりません（MUST）。

対象データの機密区分が未定義のまま、個人データ・顧客機密を扱う機能を本番環境に反映・運用してはなりません（MUST NOT）。機密区分と取り扱い基準を定義した上で進めるか、解消できない場合は人間の裁定を仰ぎ、その結果を記録します（「3. 基本原則」の原則の競合と裁定・監査証跡と整合）。

---

### Observability by Default

本番運用される機能は、観測可能（ログ・メトリクス・トレース）であるべきです（SHOULD）。

観測性の要件は後付けにせず、設計段階で考慮するべきです（SHOULD）。

詳細は standards/observability-standards.md で管理するべきです（SHOULD）。

---

### 監査証跡（Auditability）

重要な意思決定は、その作成者・承認者・根拠を追跡可能であるべきです（SHOULD）。

AIエージェントが生成または起案した変更は、AI生成であることを識別可能であるべきです（SHOULD）。例として、コミットトレーラ、PRラベル、ADR・提案の起案者欄により識別可能とします。

承認を要する行為（6章 承認マトリクス）の承認記録は、後から追跡可能な形で保持するべきです（SHOULD）。

監査証跡の具体的な記録方式は development-process.md または standards/ で管理するべきです（SHOULD）。

---

### 変更の可逆性とインシデント対応

本番に影響する変更は、ロールバックまたは復旧の手段を備えるべきです（SHOULD）。

復旧困難な不可逆操作は、設計段階でリスクを評価するべきです（SHOULD）。

インシデント対応およびロールバックの手順は development-process.md または standards/ で管理するべきです（SHOULD）。

---

### 依存関係とサプライチェーンの完全性

外部依存の追加・更新は、ライセンスおよび既知の脆弱性を確認した上で行うべきです（SHOULD）。

既知の重大な脆弱性を含む依存をマージしてはなりません（MUST NOT）。

「重大」の判定基準（深刻度スケール・閾値）は standards/security-standards.md で定義しなければなりません（MUST）。

依存関係は自動更新の仕組み（例：Renovate、Dependabot）で継続的に保守するべきです（SHOULD）。

本番に供する成果物は、その生成元（ソース）まで追跡可能であるべきです（SHOULD）。ソフトウェア部品表（SBOM）を生成し、成果物の出所（Provenance）を検証可能にするべきです（SHOULD）。

本番利用する依存・成果物は、本番利用判定（ライセンス・既知脆弱性・保守状況・出所）を通過するべきです（SHOULD）。

供給網完全性レベルの目標（例：SLSA）、SBOMの形式、出所証跡の検証方式といった具体的な機構は standards/security-standards.md および対応するADRで定義するべきです（SHOULD）。

---

### アーキテクチャの完全性（Architecture Integrity）

循環依存を導入してはなりません（MUST NOT）。

モジュール間の依存方向は明示されるべきです（SHOULD）。

アーキテクチャ境界を越える通信は、公開された契約（API・イベント等）を通じて行うべきです（SHOULD）。

具体的な境界・レイヤ構成・依存規則は architecture/* および standards/architecture-standards.md で定義するべきです（SHOULD）。安全・完全性に関わる機械検証可能な MUST / MUST NOT（循環依存の禁止等）は、対象スタックが検証に対応した時点で「8. 機械的に検証可能なルール」へ登録しなければなりません（MUST）。登録までの間は、強制台帳上で「ブートストラップ（人間レビュー）」として強制手段を割り当てます。

---

### アクセシビリティ（Accessibility）

ユーザー向けインターフェースを持つ成果物は、アクセシビリティに配慮して設計・実装するべきです（SHOULD）。

具体的な基準（WCAG 準拠レベル等）は standards/accessibility-standards.md で定義するべきです（SHOULD）。

ユーザー向けインターフェースを持たないプロジェクトには、本原則は適用しません。

---

## 4. 変更分類（Change Classification）

すべての変更を一律に扱うと、軽微な変更に過剰な負荷がかかり、重大な変更には十分な厳格さを確保できません。

AIエージェントも変更の軽重を区別できず、毎回同じ重さで判断してしまいます。

また、AI駆動開発では「通常変更」が肥大化しやすく、1行のバグ修正と大規模な新機能、内部実装変更と公開インターフェース変更が同一クラスに混在しがちです。これらは要求される厳格さ（ADR・承認・自律範囲）が異なるため、本プロジェクトでは4区分とします。

保護対象ブランチへ反映するすべての変更は、反映前にいずれかの変更クラスに分類されなければなりません（MUST。人間ゲートおよびパス対応表等の自動分類で強制）。

| クラス                              | 定義                                                   | 例                                                                                                                  |
| -------------------------------- | ---------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------ |
| Class A（統治・セキュリティ・本番重大／不可逆）       | 統治・強制機構、セキュリティ、本番障害・データに影響しうる、または不可逆な変更。最も厳格に扱う。 | 本書・standards・セキュリティ方針・CI/CD・ブランチ保護・エージェント指示ファイルの変更、認証・認可、秘密情報、スキーマ変更／マイグレーション、インフラ構成、リリース、依存の新規追加・重大更新（メジャー更新・ライセンス変更を伴うもの）、運用方針の重大な変更、本番データ操作・不可逆操作、ADR の Accepted 化 |
| Class B（アーキテクチャ・公開インターフェース）       | 構造または公開インターフェースに影響する設計変更。長期的影響を持ち、原則 ADR 化の対象。   | アーキテクチャ変更、フォルダ構成変更、フレームワーク／データベース選定、公開API・インターフェースの破壊的変更、公開APIへの後方互換な追加                                            |
| Class C（機能・不具合修正）                 | 本番挙動に影響しうるが、構造・公開IF・セキュリティには影響しない通常のコード変更。       | 後方互換の内部機能追加、不具合修正、内部リファクタリング、依存のパッチ・マイナー更新（重大更新を除く）                                                                                       |
| Class D（ドキュメント・整形）               | 本番挙動に影響しない変更（統治文書を除く。下記注参照）。                       | README・一般ドキュメント、コメント、フォーマット                                                                                        |

各クラスの具体的な判定基準（対象パスとクラスの対応表を含む）は development-process.md で定義しなければなりません（MUST）。

クラスが確定しない変更は、確定するまで Class A として扱わなければなりません（MUST）。

クラスは以下に影響します。

* ADRの要否（5章）
* 承認の要否（6章 承認マトリクス）
* 完了条件の品質ゲート（9章）
* AIエージェントの自律範囲（6章）

Class A および Class B の変更は、人間の承認なしに保護対象ブランチへ反映してはなりません（MUST NOT）。

Class C の変更は、AIエージェントが起案・準備（PR作成、テスト追加等）を自律的に実施してもよい（MAY）。

ただし保護対象ブランチへのマージは人間の承認を要します（6章参照）。

Class D の変更は、AIエージェントが起案・準備を自律的に実施してもよい（MAY）。

保護対象ブランチへのマージは原則として人間の承認を要します（6章）。

ただし、ドキュメント品質ゲート（Markdown Lint、Link Check）の全通過を条件として、AIエージェントによる Class D（統治文書を除く）の自己反映を standards/ai-governance.md で許可してもよい（MAY）。

許可する場合の条件・対象パスは同標準で定義します。

これにより、些末な変更への過剰な負荷を避けつつ、緩和の範囲を統治下に置きます。

注：変更が「ドキュメントのみ」であっても、統治・強制機構に属する文書（本書 constitution.md、governance/、standards/（ai-governance.md を含む）、エージェント指示ファイル（AGENTS.md・CLAUDE.md 等）、セキュリティ方針、CI/CD設定、ブランチ保護定義）への変更、および ADR の Accepted 化は、Class D ではなく Class A として扱います（6章・7章参照）。これらを「ドキュメント変更だから自律的に反映してよい」と解釈してはなりません（MUST NOT）。

統治の重さ（承認者数・ADR 要否の範囲・カバレッジ初期値・skills/knowledge の必須性）は、プロジェクトの規模・規制要件に応じた段階導入プロファイル（Lite / Standard / Regulated）で調整してよい（MAY。正本は development-process.md「8. 段階導入プロファイル」）。ただし本章の絶対ルール（クラス未確定は Class A として扱う等）、「6.」の作成者≠承認者・統治機構の自己反映禁止、および「8. 機械的に検証可能なルール」の安全ゲートは、いかなるプロファイルでも緩和してはなりません（MUST NOT）。

---

## 5. ADRポリシー

ADR は adr/ ディレクトリで管理しなければなりません（MUST）。

Class A または Class B に該当する設計判断（対象の定義・例示は「4. 変更分類」を正本とする）は、ADRとして記録するべきです（SHOULD）。

上記以外でも、長期的な影響を持つ設計判断はADRとして記録するべきです（SHOULD）。

判断に迷う場合はADRを作成してもよい（MAY）。

---

### ADR命名規則

ADRの命名規則（ファイル名パターン・採番方式・欠番の扱い）は、adr-rules.md で定義しなければなりません（MUST）。

adr-rules.md を命名規則の正本（SSoT）とします。

ファイル名が命名規則に準拠していることは、CI/自動検証で強制します（「8. 機械的に検証可能なルール」の機械強制ルールとして MUST）。

---

### ADRテンプレート

ADRの必須セクション（例：Status、Context、Decision、Consequences 等）および Status 管理語彙は、adr-rules.md で定義しなければなりません（MUST）。

adr-rules.md をこれらの正本（SSoT）とします。

記入用のテンプレート本体は adr-template.md として分離してもよい（MAY）。

分離する場合、adr-template.md は adr-rules.md が定める必須セクションに準拠しなければなりません（MUST）。

これにより運用規則（adr-rules.md）と記入様式（adr-template.md）の責務を分離し、単一文書への集中を避けます。

「8. 機械的に検証可能なルール」における「ADRテンプレートに必須セクションが存在すること」の検証は、adr-rules.md に定義された必須セクションを基準として行うべきです（SHOULD）。

---

### ADR Status 管理

ADRはStatusフィールドを持たなければなりません（MUST）。

Statusは adr-rules.md が定義する管理語彙のいずれかでなければなりません（MUST）。

Statusが管理語彙のいずれかであることは、CI/自動検証で強制します（「8. 機械的に検証可能なルール」の機械強制ルールとして MUST）。

あるADRが Superseded となった場合、後継のADRを参照するべきです（SHOULD）。

これによりADRが増えても決定の系譜を追跡できます。

---

### 重大な設計変更におけるADRの強制

Class A または Class B の変更を含むPRには、対応するADRへの参照、またはADRが不要である理由を記載しなければなりません（MUST）。

この要件はPRテンプレートで定型化し、CIで「いずれかが記載されていること」を検証します（「8. 機械的に検証可能なルール」に機械強制ルールとして登録・MUST）。

注：変更が重大（Class A／Class B）か否かの判定自体は人間の判断を含みます。

CIで機械的に検証できるのは「記載の有無」であり、運用上はクラス判定またはPRラベル／対象パスに基づいて本要件を適用するべきです（SHOULD）。

これにより「ADRを書いていないこと」を検知可能にし、ADRファースト原則の形骸化を防ぎます。

---

## 6. AIエージェント統治と自律境界

### 参照順序

AIエージェントは、情報を参照する際、「2. 文書管理階層」に定める優先順位に従うべきです（SHOULD）。

優先順位の正本は同章とし、本章では再掲しません。

AI運用に関する規範は、本書（原則）→ standards/ai-governance.md（詳細方針）→ エージェント実行指示（AGENTS.md および CLAUDE.md 等のツール固有ファイル）の順に詳細化されます。

下位層は上位層と矛盾してはなりません（MUST NOT）。

AIはADRに反する提案を行う場合、その理由および代替案を明示するべきです（SHOULD）。

ADRと実装内容が矛盾している場合、AIエージェントは自律的に修正してはなりません（MUST NOT）。

その場合は人間への確認を優先しなければなりません（MUST）。

---

### エージェント指示ファイルの統治

エージェント実行指示の正本は AGENTS.md とします（MUST）。

AGENTS.md は、本書（constitution.md）を上位規範として参照しなければなりません（MUST）。

CLAUDE.md 等のツール固有ファイルは、AGENTS.md への参照（ツールが対応する場合はインポート）を含む薄い委譲層とし、当該ツール固有の差分のみを記載するべきです（SHOULD）。

これにより、利用するエージェント（Claude Code、Codex、Gemini CLI、Cursor 等）がどの指示ファイルを自動読込する場合でも、本書への到達経路を保証します。

エージェント指示ファイルは統治・強制機構の一部であり、その変更は Class A として扱い、本節の承認マトリクスに従います（4章注参照）。

---

### 自律的に実施してよい行為

AIエージェントは、人間の事前承認なしに以下を起案・準備してもよい（MAY）。

これらは原則として Class D、および本番リスクの低い Class C の一部に相当します。

ただし最終的な反映（保護対象ブランチへのマージ）は本節「人間の承認を要する行為」に従います。

Class D の自己反映に関する例外は「4. 変更分類」に従います。

* ドキュメントの草案作成および編集提案
* ADRドラフト（Proposed 状態）の起票
* テストの追加・修正
* 公開インターフェースを変更しないリファクタリング（既存テストがグリーンを維持する範囲）
* バグ修正のプルリクエスト作成
* Lint / フォーマットの修正

---

### 人間の承認を要する行為（承認マトリクス）

以下の行為は、人間の承認なしに反映（保護対象ブランチへのマージ）または実環境での実行をしてはなりません（MUST NOT）。

これらは原則として Class A または Class B に相当します。

起案・準備（作業ブランチ上のファイル編集、ドラフトPRの作成）は本マトリクスの禁止対象ではなく、「自律的に実施してよい行為」および「権限・統治への変更」の定めに従います。

強制手段が機械的に整備できない行為は、必須の人間ゲート（必須レビュア・CODEOWNERS 等）で担保しなければなりません（MUST）。

| 行為                                          | 強制手段の例                     |
| ------------------------------------------- | -------------------------- |
| 保護対象ブランチへの反映（マージ。自己マージを含む。Class D の自己反映を除く） | ブランチ保護・必須レビュア              |
| ADR の Accepted 化（承認）                        | レビュー・CODEOWNERS            |
| 依存関係の新規追加、およびメジャー更新・ライセンス変更を伴う更新（その他の依存更新は Class C） | 依存スキャン・レビュー                |
| CI/CD 設定の変更                                 | CODEOWNERS・レビュー            |
| データベーススキーマ変更・マイグレーション                       | レビュー                       |
| 認証・認可・秘密情報に関わる変更                            | シークレットスキャン・レビュー            |
| 公開API／インターフェースの破壊的変更                        | レビュー                       |
| インフラ構成変更・コスト影響を伴う変更                         | レビュー                       |
| リリース（本番への反映）                                | レビュー・リリース承認                |
| 本書・standards（ai-governance.md含む）・セキュリティ方針の変更 | 「7. 変更管理」の改正手続き            |
| エージェント指示ファイル（AGENTS.md・CLAUDE.md 等）の変更        | CODEOWNERS・権限影響ラベル・レビュー       |
| 本番データに対する操作、および削除を伴う不可逆操作                   | レビュー・実行権限分離                |

注：Class D（統治文書を除くドキュメント・整形）の自己反映は、ドキュメント品質ゲートの全通過を条件として standards/ai-governance.md で許可される場合に限り、本マトリクスの例外とします（「4. 変更分類」参照）。

それ以外のすべての保護対象ブランチへの反映は人間の承認を要します。

---

### 権限・統治への変更（権限昇格の防止）

AIエージェントが自身の権限・自律範囲・実行権限を拡大し得る変更、ならびに統治・強制機構（本書、standards/ai-governance.md、エージェント指示ファイル（AGENTS.md・CLAUDE.md 等）、承認マトリクス、CI/CD設定、品質ゲート定義、ブランチ保護）に対する変更について、以下を定めます。

* AIはこれらの変更を提案してもよい（MAY）が、単独で反映してはなりません（MUST NOT）。

* AIは、提案が自身の権限・自律範囲、または統治・強制機構に影響することを明示的に開示しなければなりません（MUST）。

  これは、個々には軽微な変更（CLAUDE.md変更・CI設定変更・ADR起票など）の組み合わせによる事実上の権限昇格を防ぐためです。
* 開示をAIの自己申告のみに依存させてはなりません（SHOULD NOT）。統治・強制機構に属するパスへの変更を検出して権限影響ラベルを自動付与する等、システム側の検出を併用するべきです（SHOULD）。
* 変更の作成者と承認者は分離されなければなりません（職務分掌、MUST）。

  AIは自らが関与した権限拡大を承認または自己マージしてはなりません（MUST NOT）。
* AIエージェントは、人間の認証情報・アカウントを用いて行為してはなりません（MUST NOT）。作成者と承認者の分離を機械強制可能にするため、AIエージェントは専用の識別可能なアイデンティティ（マシンアカウント等）で行為しなければなりません（MUST）。
* 強制機構（品質ゲート、ブランチ保護等）を弱める変更を、人間の承認なしに反映してはなりません（MUST NOT）。

統治・強制機構に対応するファイル群（本書、standards/ai-governance.md、エージェント指示ファイル（AGENTS.md・CLAUDE.md 等）、CI/CD設定、ブランチ保護定義等）への変更は、CODEOWNERS／必須レビュアおよび権限影響ラベルにより、人間承認を強制しなければなりません（MUST）。

上記の各 MUST / MUST NOT は機械的に完全には検証できないため、人間ゲートで強制します。その機械検証可能な近似（統治・強制機構に属するパスへの変更PRにおける権限影響ラベルと CODEOWNERS 承認の存在）は「8. 機械的に検証可能なルール」に登録します。

---

### 自己修正ループの防止

AIエージェントは、自らの変更によって失敗した品質ゲートまたは統治・強制機構を、その失敗を回避する目的で弱める・無効化する・回避する変更を実施してはなりません（MUST NOT。人間ゲートで強制）。

回避目的の有無は機械検証できないため、本ルールは次の状態ベースのルールにより包含的に強制します：品質ゲート定義および統治・強制機構に属するパスへの変更は、その目的にかかわらず Class A として扱い、人間の承認を必須とします（MUST）。

AIエージェントが、自らの変更に起因して失敗したゲートの見直しを提案する場合、その失敗との関連を明示的に開示しなければなりません（MUST）。

失敗の正しい解消は、原因となった変更側の修正によって行うべきです（SHOULD）。

ゲート変更の作成者と承認者は分離されなければなりません（MUST）。

これは「AIの変更 → ゲート失敗 → AIがゲートを緩和 → 通過」という自己修正ループによる形式上の合格と、権限昇格とは別の経路で強制機構が骨抜きになることを防ぐためです。

### Human-in-the-Loop チェックポイント

以下のチェックポイントは義務レベルに応じて二段階に分けます（義務レベルと強制手段は独立の軸。「1.1 義務レベルと強制手段」参照）。いずれも人間ゲートで担保します。

#### 停止必須（MUST）

以下の局面では、AIは判断を停止し、人間の判断を仰がなければなりません（MUST）。これらは本書の他の箇所が MUST として定める義務に対応し、本節はその参照点です。

* 原則間の競合を自律的に解消できない時点（「3. 基本原則」の原則の競合と裁定。同節の MUST と整合）
* 本書が参照する下位文書が未整備で、判断根拠が不足する時点（「8. 機械的に検証可能なルール」ブートストラップ規定の MUST と整合）
* 機密区分が未確定のデータ等、AIへの入力可否が不明なデータを扱う時点（「データ保護とAI入力境界」と整合）
* 自身の権限、または統治・強制機構に影響する変更を提案する時点（本章「権限・統治への変更」の開示義務と整合）
* 本書・ADR・実装の間に矛盾を検出した時点（本章「参照順序」の MUST / MUST NOT と整合）

#### 停止推奨（SHOULD）

以下の局面では、AIは判断を停止し、人間の判断を仰ぐべきです（SHOULD）。これらは別途のゲート（マージ承認・品質ゲート等）が最終的な担保を与えます。

* ADR化の要否を判断する設計判断点
* 保護対象ブランチへのマージ前レビュー
* リリース前の承認
* 自らの変更が原因で失敗した品質ゲート・統治機構を変更しようとする時点（本章「自己修正ループの防止」の MUST / MUST NOT が実体を担保する）

## 7. 変更管理

### 変更対象と手続きの対応

文書の種類に応じて、改正手続きを次のように対応づけます。

これは、憲章と通常の設計変更（ADR）の中間に位置する文書（基本方針・統治機構・メタ規則）の手続きが未定義になることを防ぐためです。

| 対象                                                              | 手続き                          | 記録先         |
| --------------------------------------------------------------- | ---------------------------- | ----------- |
| 開発憲章（本書）                                                       | ガバナンス決定（下記「改正手続き」）          | governance/ |
| 基本方針（charter.md / vision.md / scope.md）                         | ガバナンス決定（下記に準ずる）             | governance/ |
| 統治・強制機構（standards/ai-governance.md、CI/CD設定、ブランチ保護定義、セキュリティ方針） | ガバナンス決定（下記に準ずる）             | governance/ |
| ADR運用規則（adr-rules.md）                                          | ガバナンス決定（下記に準ずる）             | governance/ |
| その他の技術標準（standards/* のうち上記以外）                                  | ADR（重大な場合）または通常の設計変更        | adr/        |
| 設計・実装（アーキテクチャ、コード等）                                           | ADR（必要に応じて）＋通常の変更           | adr/        |

基本方針・統治機構・メタ規則（adr-rules.md）の改正は、ADRではなくガバナンス決定として扱うべきです（SHOULD）。

これらは特定のアーキテクチャ判断ではなく、プロジェクトの統治に関わる問題だからです。

### 通常の設計変更

設計・実装に関する変更は、必要に応じてADRを作成するべきです（SHOULD）。

変更理由はADRに記録するべきです（SHOULD）。

重大な変更におけるADR参照またはADR不要理由の記載は「5. ADRポリシー」に従います。

### 開発憲章（本書）の改正手続き

本書の改正は、ADRではなくガバナンス決定として記録するべきです（SHOULD）。

改正は以下の手続きに従うべきです（SHOULD）。

* 改正は提案（Proposal）として起票する。
* 提案には変更内容・理由・影響範囲を明記する。
* 承認には、あらかじめ定めた承認者による最低レビュー数（定足数）を満たす必要がある。
* 承認者・定足数は development-process.md または governance/ で定義するべきです（SHOULD）。
* AIエージェントは本書改正を単独で承認・反映してはなりません（MUST NOT）。
* 却下された提案も記録を残すべきです（SHOULD）。

#### バージョニング方針（セマンティックバージョニング）

本書のバージョンはセマンティックバージョニング 2.0.0（`MAJOR.MINOR.PATCH`）に従わなければなりません（MUST）。形式・リセット等の一般規則は同仕様に従い、本書では再掲しません。

統治文書としての性質に合わせ、各番号の増分規則を以下のとおり定義します。本書における「公開インターフェース」とは、本書が定める規範（原則・ルール・義務）の集合を指します。

| 増分    | 規則                                                                 | 例                                          |
| ----- | ------------------------------------------------------------------ | ------------------------------------------ |
| MAJOR | 後方非互換な統治・原則の変更。既存の義務（MUST / MUST NOT）の撤廃・反転・意味変更、章や原則の削除を含む。      | 承認マトリクスの要件削除、ある原則の規範的意味の変更、Class定義の再定義     |
| MINOR | 後方互換な追加・実質的拡張。新たな原則・章・節の追加、機械検証ルールの追加、既存ガイダンスの実質的な拡張。            | 新しい原則の追加、第8章への MUST ルール追加、新たな standards の規定 |
| PATCH | 規範的意味を変えない改善。明確化・字句修正・誤記訂正・リンク修正・整形など、非規範的な洗練。                    | 表現の統一、用語の補足、参考文献URLの修正                     |

増分種別の判定が曖昧な場合は、確定前にその理由を提示するべきです（SHOULD）。

初期開発期間（Status が Draft / Proposed であり未批准の期間）は `0.y.z` を用い、この期間の規範は後方互換性を保証しません。正式批准（Status: Accepted）時に `1.0.0` とします。

レビュー中の改正提案には、プレリリース識別子（例：`1.1.0-rc.1`）を付与してもよい（MAY）。

ひとたび確定（記録）したバージョンの内容は変更してはなりません（MUST NOT）。変更が必要な場合は新たなバージョンとして発行します。

バージョン増分は改正手続き（本章）の確定結果として決定し、「13. 改正履歴」に反映するべきです（SHOULD）。採番の運用（誰がいつ確定するか等）は development-process.md または governance/ で定めるべきです（SHOULD）。

#### ガバナンス決定の Status 管理

ガバナンス決定（改正提案）は Status を持つべきです（SHOULD）。

ADRと同様に状態を管理することで、決定の系譜を追跡できます。

推奨する管理語彙は以下とします。

```text
Draft
Proposed
Accepted
Rejected
Superseded
Withdrawn
```

あるガバナンス決定が Superseded となった場合、後継の決定を参照するべきです（SHOULD）。

Withdrawn は提案者が承認前に取り下げた場合に用います。

注：ガバナンス決定の Status 語彙（本節：Draft / Proposed / Accepted / Rejected / Superseded / Withdrawn）と、ADR の Status 語彙（adr-rules.md「5. Status 管理語彙と遷移」：Proposed / Accepted / Rejected / Deprecated / Superseded）は、別アーティファクトの別語彙です。両者を混同してはなりません（MUST NOT）。ガバナンス決定は Draft / Withdrawn を持ち Deprecated を持たず、ADR は Deprecated を持ち Draft / Withdrawn を持ちません。

改正提案および確定記録は governance/ で管理するべきです（SHOULD）。例：

```text
governance/
├─ proposals/   改正提案（Proposal）
└─ decisions/   改正の確定記録（承認・却下を含む）
```

改正が確定した場合、その内容を「13. 改正履歴」に記録するべきです（SHOULD）。

「13. 改正履歴」は追跡用の派生サマリであり、ガバナンス決定の正本（SSoT）は governance/decisions/ です。

本書自身が変更履歴を追跡可能であることは、Documentation as Code 原則との整合のために必要です。

### 緊急時の例外（Break-glass）

本番障害対応等の緊急時に限り、あらかじめ指名された緊急承認者の承認により、「8. 機械的に検証可能なルール」の品質ゲートの一部を事前検証から事後検証へ切り替えてもよい（MAY）。

緊急例外は、人間の承認そのものを免除しません。緊急時であっても、人間（緊急承認者）の承認なしに保護対象ブランチへ反映してはなりません（MUST NOT）。

緊急例外を適用した場合、適用の事実・理由・範囲・承認者を記録し、72時間以内に事後レビュー（スキップしたゲートの事後実行を含む）を完了しなければなりません（MUST）。

緊急例外を、統治・強制機構の恒久的な変更・緩和、または本章の改正手続きの回避に用いてはなりません（MUST NOT）。

緊急承認者の指名および緊急時手順の詳細は development-process.md または governance/ で定義するべきです（SHOULD）。

見直しの結果「変更不要」であった場合も、見直しを実施した記録を残すべきです（SHOULD）。

## 8. 機械的に検証可能なルール

本章は、機械強制（CI/CD または自動検証）により反映前に強制するルールを登録します。

本章に登録した MUST および MUST NOT は、CI/CD または自動検証によって実際に強制されていなければなりません（MUST）。

これにより MUST が定義上の「強制可能」にとどまらず、実際に強制されている状態を保証します。

本章の各ルールには、強制手段（構造的強制／機械強制／人間ゲート）と整備状況（整備済み／ブートストラップ）を対応づけた強制台帳を維持します。強制台帳の正本は governance/enforcement-ledger.md とし、本書中の各 MUST / MUST NOT に強制手段（構造的強制／機械強制／人間ゲート／ブートストラップのいずれか）を割り当てます（MUST）。

台帳は、形骸化を避けるため、可能な限り本書の MUST / MUST NOT 抽出から生成し、手動同期を最小化するべきです（SHOULD）。台帳の網羅性（本書のすべての MUST / MUST NOT が強制手段の割当を持つこと）は、「7. 変更管理」の定期見直しにおける必須確認項目とするべきです（SHOULD）。検証手段が整備された範囲は、人間レビューから自動検証へ移行するべきです（SHOULD）。

人間ゲートで担保する MUST / MUST NOT は、本章ではなく「6. AIエージェント統治と自律境界」の承認マトリクスに登録します。

CI/CDの整備は優先的に行うべきです（SHOULD）。

検証手段が整備された時点で、当該ルールは人間レビューから自動検証へ移行するべきです（SHOULD）。

検証手段が未整備のMUST/MUST NOTを、整備済みであるかのように扱ってはなりません（MUST NOT）。

本書が参照する下位文書（development-process.md、adr-rules.md、standards/* 等）が未整備の場合、当該事項は「未定義」として扱います。AIエージェントは未定義の事項について自律判断してはならず、人間の判断を仰がなければなりません（MUST）。変更クラスの判定基準が未整備の間も、フェイルセーフ既定（クラス未確定の変更は Class A として扱う）は適用されます。

## 9. 完了条件（Definition of Done）

完了条件は「必須の品質ゲート」と「クラス固有の条件」で構成されます。

品質ゲートの具体的な項目は「8. 機械的に検証可能なルール」を正本（SSoT）とし、本章では再掲せず、クラスへの適用と設計上の条件のみを定めます（重複を避けるため）。

完了条件 = 8章の必須品質ゲート ＋ クラス固有条件

以下は必須とします（MUST）。

* Class A／B／C は、「8. 機械的に検証可能なルール」のコード品質・セキュリティに関する必須ゲート（ビルド・型チェック・自動テスト・カバレッジ・シークレットスキャン・脆弱性スキャン等）をすべて通過しなければなりません（MUST）。
* Class D（本番挙動に影響しない変更）は、ドキュメント品質ゲート（Markdown Lint、Link Check）を通過しなければなりません（MUST）。変更がコードファイルに触れる場合（コメント・フォーマット等）は、加えてビルド・コード Lint・既存テストの通過を維持しなければなりません（MUST）。コードファイルに触れない変更に、コード品質ゲートは適用しません。
* 重大な変更（Class A／Class B）を含むPRには、ADRへの参照またはADR不要理由が記載されていなければなりません（MUST。5章参照）。
* 品質ゲートを通過していない変更を保護対象ブランチへ反映してはなりません（MUST NOT。8章参照）。

加えて、設計変更は以下を満たすべきです（SHOULD）。

* 必要なADRが作成されている、またはADR不要理由が記載されている
* 関連ドキュメントが更新されている
* エージェント指示ファイル（AGENTS.md・CLAUDE.md 等）および standards/ai-governance.md が更新されている（必要な場合）
* 変更が重大な性能劣化を導入していない（standards/performance-standards.md に性能目標・測定基準が定義されている場合、それに従って確認する）
* 実装が完了している
* テストが完了している
* レビューが完了している
* 監査証跡（作成者・承認者・根拠）が追跡可能である
* CIが成功している
