# ADR運用規則（ADR-RULES）

* Version: 0.1.0（Proposed / ドラフト）
* Date: 2026-04-01
* Status: Proposed
* 上位規範: constitution.md（開発憲章）
* 付属文書: adr-template.md（完全プロファイル様式）／ adr-template-minimal.md（最小プロファイル様式）

---

## 目的と位置づけ

本書は、ADR（Architecture Decision Record）の運用規則——命名規則、プロファイル、必須セクション、フロントマター、Status 管理語彙とその遷移——の正本（SSoT）です。

本書は constitution.md（開発憲章）に従属するメタ文書であり、個別 ADR の構造的事項には優先します（憲章「2. 文書管理階層」参照）。

ADR 運用の導入とフォーマット標準の選定という意思決定の経緯は、ADR-0000 に記録されます。運用規則の正本は本書であり、ADR-0000 の付録等に規約の記載が存在する場合も、本書が優先します（重複する規約は本書への参照に置き換えるべきです（SHOULD））。

本書の改正は、ADR ではなくガバナンス決定として扱います（憲章「7. 変更管理」参照）。

RFC キーワード（MUST / SHOULD / MAY 等）の解釈、およびキーワードを含まない記述が非規範であることは、憲章「1. RFC 2119 / RFC 8174 準拠」に従います。

## 1. 命名規則

ADR のファイル名は、次の形式に従わなければなりません（MUST）。

```text
adr-NNNN-short-title.md
```

* `NNNN` は `0001` から始まる4桁ゼロ埋めの連番としなければなりません（MUST）。`0000` は ADR 運用自体の導入決定（ADR-0000）のために予約します。
* `short-title` は小文字の英数字をハイフンで連結した短い識別名（kebab-case）とするべきです（SHOULD）。
* 表記規則：ファイル名は小文字 `adr-NNNN` とし、フロントマター `id`・本文見出し（H1）・ヘッダ表では大文字 `ADR-NNNN` と表記します。番号部（`NNNN`）は、ファイル名・`id`・本文見出し・ヘッダ表で完全に一致しなければなりません（MUST。「4. フロントマター規約」参照）。

例：

```text
adr-0000-adr-format-and-governance.md
adr-0001-database-selection.md
adr-0002-authentication-strategy.md
```

機械検証用の参照正規表現：

```text
^adr-[0-9]{4}-[a-z0-9]+(-[a-z0-9]+)*\.md$
```

採番・名前空間・欠番の扱い：

* 番号は作成順の連番とし、一度使用した番号を再利用してはなりません（MUST NOT）。欠番は許容します（番号の詰め直しは行いません）。
* 採番の名前空間は、既定としてリポジトリ内で一意とします。**プレフィックスによる名前空間の区別は将来拡張であり、既定では使用しません（YAGNI。現時点で複数案件の分離要件がないため）。** 顧客別・案件別の分離が現に必要となった時点で本規則を改正してプレフィックスを採用してよい（MAY。例：ファイル名 `adr-prjx-0001-…`、`id` は `ADR-PRJX-0001`）。採用する場合、参照正規表現を対応形式へ拡張し、名前空間の一意性の範囲（全社一意か適用スコープ単位か）を当該改正で確定します。
* 不採用・廃止となった ADR も、ファイルと番号を保持し、削除ではなく Status（Rejected / Deprecated / Superseded）で表現するべきです（SHOULD）。これは憲章の基本原則「監査証跡」「変更の可逆性」に基づきます。

格納場所は adr/ ディレクトリとします（正本：憲章「5. ADRポリシー」。本書は再定義しません）。

ファイル名が本章の命名規則に準拠していることは、CI/自動検証で強制します（憲章「8. 機械的に検証可能なルール」）。

## 2. プロファイル

ADR は、最小（minimal）または完全（full）のいずれかのプロファイルを選択しなければなりません（MUST）。選択はフロントマターの `profile` に記録します。

* 完全プロファイルを用いるべき場合（SHOULD）：適用スコープが organization ／ division ／ product の決定、Class A に相当する決定、監査要件が強い決定、複数選択肢の定量比較を要する決定。
* 上記以外（目安：project ／ department 粒度、小規模組織）は最小プロファイルを用いてもよい（MAY）。
* レビューアは、必要に応じて完全プロファイルへの引き上げを求めてもよい（MAY）。

各プロファイルの必須セクションは「3. 必須セクション」に定めます。

---

## 3. 必須セクション

ADR の機械可読メタデータはフロントマターを唯一の正本（SSoT）とし、本文には再掲しません（ヘッダ表・ステータスのチェックボックス等を設けない）。これにより本文⇄フロントマターの二重管理を構造的に排します（憲章「3. 基本原則／SSoT」）。

### 両プロファイル共通の必須セクション（MUST）

タイトル（H1。`ADR-NNNN: 決定の短い表題` 形式）／変更履歴／コンテキスト／意思決定事項／選択肢／決定（案）／承認／結果

### 完全プロファイルでの追加必須セクション（MUST）

適用スコープ／評価観点／検討項目／評価／関連ADR／参考資料

### 任意セクション（MAY）

決定要因（最小プロファイルでは、評価観点を省略するため記入を推奨します（SHOULD））／適用スコープ（最小プロファイルでは省略してよい。スコープはフロントマター `scope` を正本とする）

ステータスは本文に専用セクションを設けず、フロントマター `status` を正本とし、遷移の証跡は「変更履歴」に記録します（「5. Status 管理語彙と遷移」）。関係 ADR は完全プロファイルの「関連ADR」表に転記してよいが、正本はフロントマターの関係キーであり、一覧と逆方向の関係は自動生成索引（本章「索引」）が担います。

セクションの正準見出し・順序・記入ガイド（スコア記法・指示文の扱い等の記入様式規約を含む）は adr-template.md を正とします。

古典的な ADR 要素との対応（非規範の参考）：Status＝フロントマター `status`、Context＝コンテキスト、Decision＝決定（案）および承認、Consequences＝結果。

必須セクションの存在は、必須見出しの存在検証およびフロントマター検証により、CI/自動検証で強制します（憲章「8. 機械的に検証可能なルール」）。

## 4. フロントマター規約

各 ADR は、冒頭に YAML フロントマターを持たなければなりません（MUST）。フロントマターは機械可読メタデータの正本（SSoT）です。

フロントマターは機械可読メタデータの唯一の正本であり、同じ値を本文へ再掲しません（ヘッダ表・ステータスのチェックボックスは設けない。「3. 必須セクション」参照）。完全プロファイルの「関連ADR」表は人間可読の補助であり、設ける場合は対応する関係キーと一致させなければなりません（MUST）。一覧および逆方向の関係の正本は自動生成索引が担います（本章「索引」）。

監査上の承認証跡は、「承認」セクションおよび「変更履歴」の記載によります。フロントマターの値のみをもって承認の証跡としてはなりません（MUST NOT）。

### 必須キー（MUST）

`id` ／ `title` ／ `status` ／ `date` ／ `last_updated` ／ `profile` ／ `scope` ／ `proposer` ／ `decision-makers` ／ `consulted` ／ `review_after` ／ `depends_on` ／ `supersedes` ／ `superseded_by` ／ `relates_to`

必須キーは「キーの存在」が必須です。次のキーの値は空でもよい（MAY）：関係キー（`depends_on` ／ `supersedes` ／ `superseded_by` ／ `relates_to`）、`consulted`、`informed`、ならびに**承認段階に依存する `decision-makers` と `review_after`**。後二者は `status` が `accepted` の場合は非空でなければなりません（MUST。本章「機械検証可能な制約」参照）。これは Proposed 段階の ADR（承認者・見直し期日が未確定）が検証で不合格にならないようにするためです。

### 任意キー（MAY）

`tags` ／ `domain` ／ `systems` ／ `risk` ／ `informed`

`status` ／ `date` ／ `decision-makers` ／ `consulted` ／ `informed` は MADR 互換キーです（非規範の参考）。

表記規則：MADR 互換キー（`decision-makers` 等）は MADR に合わせてハイフン区切り、プロジェクト固有キー（`last_updated` ／ `review_after` ／ `depends_on` ／ `superseded_by` ／ `relates_to` 等）は snake_case です。キー名は上記「必須キー」「任意キー」の一覧の表記どおり（綴り・区切り文字を含む）に記載します。必須キーの存在検証（本章「機械検証可能な制約」）はこの表記で照合します。

Status が Superseded の ADR では、`superseded_by` に後継 ADR を記載しなければなりません（MUST）。

### 機械検証可能な制約（MUST）

* `id`（`ADR-NNNN`）の番号部が、ファイル名（`adr-NNNN-…`）の番号部と一致すること
* `status` が「5. Status 管理語彙と遷移」の語彙（小文字表記）のいずれかであること
* `date` ／ `last_updated` が `YYYY-MM-DD` 形式であること
* `review_after` は、`status` が `accepted` の場合に限り、非空かつ `YYYY-MM-DD` 形式であること（Proposed／Rejected 段階では空文字 `""` を許容する）
* `decision-makers` は、`status` が `accepted` の場合に限り、非空であること
* `profile` が `minimal` または `full` であること
* `scope` が `organization` ／ `division` ／ `department` ／ `product` ／ `project` のいずれかであること

### 索引

ADR の件数増加に備え、フロントマターから索引（ADR 一覧・関係グラフを含む）を自動生成し、`adr/INDEX.md` として維持するべきです（SHOULD）。生成は `scripts/generate_adr_index.py` 等で行い、CI で再生成して差分が無いことを検証します（憲章「8. 機械的に検証可能なルール」・SSoT 原則）。

索引は手動編集しない派生サマリであり、正本は各 ADR のフロントマターです。

ある ADR が Accepted となった後に、後続 ADR との関係（被参照・被置換などの逆方向の関係）を表現する必要が生じた場合、当該 Accepted ADR の本文・フロントマターを改変せず、索引側で関係を集約します。これにより Accepted の不変性（「5. 遷移と不変性の規則」）と関係の追跡可能性を両立させます。

## 5. Status 管理語彙と遷移

Status は、以下の管理語彙のいずれかでなければなりません（MUST）。

| Status | 意味 |
| --- | --- |
| Proposed | 提案中。レビュー・議論の対象 |
| Accepted | 承認済み。現行の有効な決定 |
| Rejected | 不採用。決定として採用されなかった |
| Deprecated | 廃止予定または廃止済み。後継の決定を持たない |
| Superseded | 後継の ADR により置き換えられた |

表記規則：正準語彙は上表のキャピタライズ表記とします。フロントマターの `status` は小文字（`proposed` 等）で記録しなければなりません（MUST）。CI 照合は小文字化した完全一致で行います。

### Status 遷移

許容される遷移は以下のとおりです。これ以外の遷移をしてはなりません（MUST NOT）。

```text
Proposed → Accepted | Rejected
Accepted → Deprecated | Superseded
```

### 遷移と不変性の規則

* Status を変更した場合、「変更履歴」への記録（日付・変更者・遷移内容・理由）と、フロントマターの同時更新を行わなければなりません（MUST）。
* Superseded とする場合、後継 ADR への参照を本文および `superseded_by` に含めなければなりません（MUST）。
* Accepted 以降、本文の実質的な変更をしてはなりません（MUST NOT）。内容の変更が必要な場合は新しい ADR を起案し、当該 ADR を Superseded とします。誤記修正等の軽微な変更は、変更履歴に記録した上で行ってもよい（MAY）。
* Rejected ／ Deprecated ／ Superseded の ADR は、記録として保持し、削除しないべきです（SHOULD）。

### 承認との関係

Status を Accepted へ変更する行為は、憲章の承認マトリクスに従います（Class A・人間の承認必須。憲章「4. 変更分類」「6. AIエージェント統治と自律境界」参照）。本書はこの義務を再定義せず、参照のみを行います。

## 6. テンプレートの管理と改訂

記入様式の正本は、完全プロファイルが adr-template.md、最小プロファイルが adr-template-minimal.md です。最小プロファイルでは adr-template-minimal.md を用いてよい（MAY）。

両テンプレートは、本書の必須セクション（3章。プロファイル別）およびフロントマター規約（4章）に準拠しなければなりません（MUST。憲章「5. ADRポリシー」参照）。

adr-template.md の改訂は、同ファイル内の「テンプレート変更履歴」に記録しなければなりません（MUST）。

必須セクション・必須キー・Status 規約に関わるテンプレートの変更は、本書の改正（ガバナンス決定）を伴わなければなりません（MUST）。記入ガイド等の文言改善は、変更履歴への記録の上で行ってもよい（MAY）。
