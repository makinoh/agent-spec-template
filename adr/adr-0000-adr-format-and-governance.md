## コンテキスト

意思決定が必要となった背景・前提・制約を客観的に整理する。本セクションは状況把握のための共通前提であり、判断そのものを誘導しない。

### 背景

- アーキテクチャ上の重要な決定の経緯が議事録・チケット・口頭伝達等に散在すると、追跡・監査・引き継ぎのコストが増大することが一般に知られている。本ADRは、その対策として ADR（Architecture Decision Record）を導入するか否か、導入する場合にどの形式を標準とするかを判断するために起案された。
- ADRは、アーキテクチャ上の重要な決定とその理由・影響を記録し、将来の関係者が決定の経緯を追跡できるようにするための文書である。

### 前提（Assumptions）

- 決定の記録形式自体もガバナンス対象となり得る。形式の統一は監査性・再利用性・オンボーディング効率に影響する。
- 機械可読な構造化（メタデータ・見出し構造）は、検索・自動集計・生成AIによる活用の前提条件となる。
- 本ADRは「意思決定の支援」を目的とし、「意思決定の代行」は行わない。特定案の事前採用、結論の決め打ち、推奨案の固定化は行わない。最終判断はレビューアおよび承認者に委ねる。
- 本ADR自体は暫定形式（選択肢Cに相当する内製構成）で記述している。本決定により選択肢A・B等の別形式が採用された場合の本ADRの取り扱い（採用形式への改稿、または初版の歴史的記録としてのそのまま保存）は、承認時に確定する。

### 制約（Constraints）

- 本決定に適用される組織固有の制約（既存の文書標準・契約上の文書要件・利用可能なツールチェーン等）: [起案者が組織の実情に応じて記入。該当がなければ「特になし」と記す]

### 記入負荷の方針（最小化）

新規の**自由記述**が必要となるのは、原則として「評価」「決定（案）」「結果」の3セクション、および「承認」の成功条件・確認方法欄（任意）のみである。それ以外のセクションは、選択（チェックボックス）・転記（日付・氏名・番号）・確認（チェックリスト）で完了するよう設計している。

なお本ADRでは、「意思決定の問い」「選択肢」「評価観点」は起案時に記載済みである。汎用テンプレート（`adr-template.md`）から新規ADRを起案する場合は、これらのセクションも起案者が記述する（記入分担はテンプレート冒頭に明記）。

## 意思決定事項

本ADRで決定すべき問いを明確化する。

- 決定の問い: ADR の**導入自体は constitution.md「5. ADRポリシー」で既に確定済み（MUST）**であるため、本ADRは「アーキテクチャ上の重要な意思決定の記録に、どのフォーマットを標準採用するか」を決定する。
- 決定の範囲（含む）: 標準フォーマットの選定／ADR運用規則（採番・命名・格納・プロファイル・ステータス運用・フロントマター。正本：adr-rules.md）の確定
- 決定の範囲（含まない）: ADR を導入するか否か（憲章で確定済み）／個別のアーキテクチャ決定の内容そのもの／文書管理ツール・リポジトリ製品の選定／RFC・設計仕様書等の関連文書プロセスの設計（必要な場合は後続ADRで扱う）
- ブートストラップ手続き: 本ADRは ADR 運用の起点であるため、**最初に Accepted 化**して adr-rules.md の運用規則を発効させる。これ以降の ADR は本ADRが定めた形式に従う。
- 期限・タイムボックス: YYYY-MM-DD ※起案者が記入
- 未解決の問い: 本ADR自身の形式の取り扱い（採用形式への改稿か、初版の歴史的記録としての保存か。コンテキスト「前提」参照）は承認時に確定する。格納先は constitution.md「5. ADRポリシー」（adr/）、フロントマター整合検査は adr-rules.md「4. フロントマター規約」および constitution.md「8. 機械的に検証可能なルール」により定義済み。採番名前空間のプレフィックスは将来拡張であり既定では使用しない（adr-rules.md「1. 命名規則」）。

## 評価観点

各選択肢を比較するための観点と定義。**スコアは1〜5の数値（5が最良）を標準記法とする。** 重み付けを行う場合は重みを1〜3の数値で設定し、加重合計（スコア×重みの総和）で比較できるようにする。記号（◎/○/△/×）は重み付けを行わない場合に限り使用してよい。重みの設定および採点・所見はレビューアに委ね、本ADRは値を事前に埋めない。

| 観点 | 定義（何を評価するか） | 重み（任意、1〜3） |
|------|------------------------|--------------------|
| 導入コスト | 初期整備（テンプレート作成・教育・移行）に要する工数・費用 | |
| 運用コスト | 継続運用（記入・レビュー・維持）に要する工数・費用 | |
| 学習コスト | 関係者が形式を習得し記入できるまでの負荷 | |
| ガバナンス適合性 | 適用スコープ・承認フロー・階層継承を表現できる度合い | |
| 監査適合性 | 履歴・承認・変更追跡が監査要件を満たす度合い | |
| AI活用適合性 | 機械可読性・構造化により生成AI/検索/自動分析へ供しやすい度合い | |
| スケーラビリティ | 件数増加・組織拡大・多階層化に耐える度合い | |
| 保守性 | テンプレート・記録の長期的な維持容易性 | |
| 互換性・エコシステム | 既存ツール・外部事例・教育リソースとの親和性 | |

> 必要に応じて観点を追加してよい（例：セキュリティ適合性、規制対応、国際化対応など）。

## 検討項目

意思決定に際して確認すべき論点のチェックリスト。客観的な確認事項であり、判断結果は含まない。

- [ ] 適用スコープと上位・下位ガバナンス方針との整合は確認したか
- [ ] 既存の社内標準・既存ADR資産との関係を確認したか
- [ ] 監査・コンプライアンス要件（保持期間、承認証跡）を確認したか
- [ ] ツールチェーン（リポジトリ、CI、文書管理）との連携要件を確認したか
- [ ] 記入負荷と運用継続性のバランスを確認したか
- [ ] 機械可読性・AI活用の必要性を確認したか
- [ ] フロントマター等の機械可読メタデータについて、整合検査（CI・リント等）の実現手段を確認したか
- [ ] 移行が必要な場合の既存記録の取り扱いを確認したか
- [ ] （受託・常駐の場合）顧客・発注者の指定標準（文書規約・テンプレート・納品要件）との優先関係を、契約・SOW に照らして確認したか
- [ ] （受託・常駐の場合）ADRを顧客への成果物・納品物とするか、機密区分と開示範囲（社内限り／顧客共有／公開）を確認したか
- [ ] （受託・常駐の場合）顧客別・案件別の採番名前空間や保管場所の分離の要否を確認したか

> 受託・客先常駐の形態に該当しない場合、末尾3項目は「該当なし」としてよい。

## 選択肢

各選択肢のメリット・デメリットは客観的・一般的事実として記載する。本ADRは特定案を推奨しない。

### 選択肢 A：Nygard ADR

Michael Nygard が2011年に提唱した最小構成のADR。基本項目は Title / Status / Context / Decision / Consequences。

**メリット**

- 項目が5要素と少なく、1件あたりの記入負荷が低い。
- 提唱から実績が長く、解説記事・公開事例・テンプレートが豊富で、習得とオンボーディングが容易。
- 構造が単純なため、ツールやチーム間での移植・互換が容易で、属人化しにくい。
- 軽量であるため件数が増えても運用負荷が線形に増えにくい。

**デメリット**

- 複数案を横並びで比較する項目や評価観点欄を標準で持たないため、比較検討の記録には追記の工夫が必要。
- 代替案・トレードオフを構造化して残す標準フィールドがない。
- 適用スコープ・承認者・階層継承などのガバナンス用メタデータを管理する仕組みが標準では不足し、大規模・多階層運用では拡張が必要。

### 選択肢 B：MADR（Markdown Architectural Decision Records）

複数案比較と意思決定根拠の構造化を重視した、Markdownベースの公開テンプレート（2017年初版）。任意のYAMLフロントマター（標準キーは status / date / decision-makers / consulted / informed の5種）によるメタデータ管理を備える。v3.0（2022年）で正式名称を「Markdown **Any** Decision Records」へ一時改称したが、v4.0.0（2024年）で「Markdown **Architectural** Decision Records」へ回帰した（任意の決定にも適用可能としつつ、焦点をアーキテクチャ決定に置く整理）。完全版・最小版・bare版のテンプレートが配布されている。

**メリット**

- 「検討した選択肢」「決定要因」「各案の長所短所（Pros and Cons）」を記録する項目を標準で備え、比較検討の証跡が残しやすい。
- フロントマターによりステータス・日付・決定者などのメタデータを機械可読に管理でき、検索・自動集計・AI活用に適する。
- 決定の実装・遵守をどう確認するかを記す「Confirmation」節を標準で備え、決定後の検証方法まで記録できる。
- Markdownベースでバージョン管理システムと親和性が高く、公式テンプレート（完全版・最小版・bare版）が配布されている。
- 構造が明確なため、監査時に決定根拠を追跡しやすい。

**デメリット**

- 完全版は項目数が多く、Nygard比で1件あたりの記入負荷が高くなりやすい。
- フロントマター項目やセクション運用のルールを組織で統一・維持する必要がある（標準キーは上記5種のみで、適用スコープ・承認フロー・変更履歴等のガバナンス項目は組織側での拡張が必要）。
- 提唱（2017年）からの歴史がNygard比で浅く、既存にNygard等の資産がある場合は移行コストが生じる。

### 選択肢 C：独自フォーマット

自組織のガバナンス要件に合わせて項目を定義する内製テンプレート。本ADRの暫定形式および付属テンプレート（`adr-template.md`）はこれに相当する。ゼロからの独自設計ではなく、コンテキスト／検討した選択肢／決定／各案のメリット・デメリットというMADR系テンプレートと共通の骨格に、適用スコープ・承認・変更履歴・プロファイル等のガバナンス層を加えた拡張構成をとり、フロントマターのキーもMADR互換キーを含む（adr-rules.md「4. フロントマター規約」）。外部標準との互換性の程度は、この構成・キーの共有度合いに依存する。

**メリット**

- 適用スコープ・承認フロー・監査項目・AI活用観点など自組織固有の要件に合わせて項目を最適化でき、不要項目を削って記入負荷を抑えられる。
- 既存の社内文書管理・審査プロセス・ツールへ直接組み込みやすい。
- 階層型ガバナンス（Organization→Project の継承）など独自の統制構造を素直に表現できる。

**デメリット**

- テンプレートの設計・改訂・教育を自組織で継続的に担う必要があり、保守コストが発生する。
- 外部事例・既製ツール・教育コンテンツとの互換性は標準テンプレート採用時より低くなり得る（低下の程度は標準との構成・キーの共有度合いに依存する）。オンボーディングや採用支援を外部資料に頼りにくい。
- 標準準拠を対外的に示しにくく、監査や説明責任の場面で裏付けが弱くなる場合がある。
- 設計品質が属人化しやすく、設計者が変わると一貫性が失われるリスクがある。

### 選択肢 D：現状維持（ADRを標準導入しない）

専用のADRフォーマットを定めず、既存の記録手段（設計書、議事録、Wiki、チケット等）による運用を継続する。**本選択肢は constitution.md「5. ADRポリシー」（ADR は adr/ で管理しなければならない＝MUST）と矛盾するため、実際には採用できない。比較のための理論的ベースラインとしてのみ記載する。**

**メリット**

- 導入・教育・移行のコストが発生せず、既存の業務フローを変更する必要がない。
- 記録の形式・粒度をチームや案件ごとの裁量に委ねられる。
- 既存ツール（Wiki・チケット管理等）への投資をそのまま活かせる。

**デメリット**

- 決定の経緯・理由・却下案が複数の媒体に散在しやすく、追跡・監査・オンボーディングのコストが恒常的に発生する。
- 記録の有無・品質が担当者に依存し（属人化）、組織として決定知識を蓄積しにくい。
- 構造化・機械可読性がないため、検索・自動集計・生成AIによる活用に供しにくい。

> 必要に応じて選択肢を追加してよい（例：Y-Statements、ハイブリッド運用など）。
