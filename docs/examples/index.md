# 実例で学ぶ

テンプレートには、**書き方の手本**になるサンプルが同梱されています。
ここでは「どこを見れば何が学べるか」を案内します。

> これらは **GDPR を題材にしたサンプル**です。実プロジェクトでは置き換えます（残しておく必要はありません）。

## サンプル一覧

| 場所 | 何の手本か | 学べること |
| --- | --- | --- |
| `specs/001-user-profile-export/` | 良い **spec** | What/Why の限定、テスト可能な FR、機密区分 |
| `specs/002-account-deletion/` | **不可逆操作**を含む spec | Class A の慎重さ、削除の扱い |
| `adr/adr-0000-adr-format-and-governance.md` | 良い **ADR**（full） | 選択肢の客観比較、結論を決め打ちしない姿勢 |

## 例1 — `specs/001`：ユーザープロフィールのエクスポート

GDPR 第20条（データポータビリティ）を題材に、「自分のプロフィールを機械可読形式で取得できる」機能の仕様です。

**ここを見る:**

- **非ゴールの明記** — 「他ユーザのデータ取得」「管理者の一括エクスポート」「アカウント削除」を**やらないと宣言**。
- **テスト可能な FR** — `FR-3: 要求者自身のデータのみを返し、他ユーザのデータを返してはならない（MUST NOT）`。曖昧語がない。
- **ユーザーシナリオ** — `US-3` で「他人のデータは取得できない」を Given/When/Then で受け入れ基準化。
- **データ機密区分** — プロフィール = 個人データ（PII）= **Restricted**。本番 PII を AI に渡さない、と spec に明記。
- **Constitution Alignment** — What/Why に限定、機密区分を特定、暫定 Class B、設計判断は ADR-0001 へ、と自己点検。

> **学び:** spec は「何を・なぜ」に徹し、設計（How）も設計理由（Why-this-design）も持ち込まない。
> 認可（自分のデータのみ）はセキュリティの中核なので **Class A 相当**として扱う。

## 例2 — `specs/002`：アカウント削除

「永続データの不可逆操作（削除）」を含む題材です。読み比べると、**可逆性とクラス**の違いが分かります。

**ここを見る:**

- 削除は**取り消せない**操作 → リスク評価と慎重なクラス判定が要る。
- 「論理削除か物理削除か」「猶予期間」などは ADR で設計理由を残す対象。

> **学び:** 同じ「ユーザーデータを扱う機能」でも、**読み取り（001）と不可逆削除（002）でリスクが段違い**。
> 変更クラスと承認の重さが変わる。

## 例3 — `adr-0000`：ADR 運用自体の決定（full プロファイル）

「ADR をどの形式で運用するか」を決めた、**最初の ADR**（番号 0000 は予約）。full プロファイルの手本です。

**ここを見る:**

- **フロントマターが正本** — `id` / `status` / `profile: full` / `scope` / `decision-makers` などを機械可読に管理。本文に二重記載しない。
- **選択肢の客観比較** — Nygard / MADR / 独自 / 現状維持 を、メリット・デメリットを**中立に**列挙。
- **結論を決め打ちしない** — 「評価」「決定（案）」「承認」を分け、加重合計だけで機械的に決めない。承認をもって確定。
- **ブートストラップ** — この ADR を**最初に Accepted 化**して `adr-rules.md` の運用規則を発効させる、という起点設計。

> **学び:** ADR は「採用案の宣言」ではなく「**選択肢と理由の記録**」。却下案を残すことに価値がある。

## サンプルから自分のプロジェクトへ

1. サンプルを読んで粒度と書き方をつかむ。
2. 自分の機能で [チュートリアル3](../tutorials/03-write-spec.md)（spec）と [チュートリアル2](../tutorials/02-write-adr.md)（ADR）を実践。
3. サンプルは置き換えるか、`specs/` から削除する。
