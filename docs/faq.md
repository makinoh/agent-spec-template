# よくある質問（FAQ）

初学者が抱きやすい疑問をまとめました。困ったら [トラブルシューティング](troubleshooting.md) と [用語集](reference/glossary.md) も併用してください。

## 基本・全体像

**Q1. これは何ですか？**  
AI エージェントに、安全に・追跡できる形でコードを書かせるための **開発プロセスの雛形**であり、その**運用方法を学ぶ教材**です。実行アプリのコードは含みません。

**Q2. フレームワークやライブラリですか？**  
いいえ。実行コードを含まない「開発の進め方」の雛形です。あなたのアプリのコードを足して使います。

**Q3. 何を解決しますか？**  
AI に任せると失われがちな「設計理由・承認記録・データ安全性」を、ルールと CI で**仕組みとして**守ります。詳しくは [ホーム](index.md)。

**Q4. 対応する言語・スタックは？**  
不問です。CI は Node / Python / Go / Java を自動検出します。コードが無い段階でも CI はグリーンで通ります。

**Q5. GitHub は必須ですか？**  
CI は GitHub Actions を前提にしますが、考え方は他の CI にも移植できます。設定手順は `ADOPTION.md`。

**Q6. 費用はかかりますか？**  
テンプレート自体は MIT ライセンスで無料です。AI ツールや CI の利用料は各サービスに従います。

**Q7. 個人開発には重すぎませんか？**  
必要な分だけ採用できます。[Lite プロファイル](governance/index.md)で軽く始め、後から強化できます。絶対ルール（本番データを AI に渡さない等）だけは常に有効です。

**Q8. 既存プロジェクトに後から導入できますか？**  
できます。基本方針・ADR・ガバナンスから段階的に取り入れ、CI ゲートを徐々に有効化します。

**Q9. どこから読めばいいですか？**  
[学習ロードマップ](learning-path.md)（Day 1〜3）が最短です。まず触りたいなら [30分クイックスタート](getting-started/quickstart.md)。

**Q10. 学ぶのにどれくらいかかりますか？**  
全体像なら 1 日、実践込みで 3 日が目安です（[学習ロードマップ](learning-path.md)）。

## AI 駆動開発・安全性

**Q11. AI が勝手に `main` にマージしますか？**  
しません。保護ブランチへの反映は人間承認が必須です（軽微なドキュメントを除く）。AI は自分が関わった権限拡大を承認できません。

**Q12. 本番データを AI に渡しても大丈夫ですか？**  
**絶対に渡しません（MUST NOT）。** 本番の個人データ・顧客機密・秘密情報を AI / 外部 AI に入力してはいけません。開発・テストは合成データを使います。

**Q13. なぜ「渡さない」を最優先にするのですか？**  
一度外部 AI へ送った機密は取り消せないからです。本テンプレートは「気をつける」ではなく、**そもそも本番データに到達できる権限を AI に与えない**（構造的強制）ことを第一とします。

**Q14. どの AI ツールが使えますか？**  
Claude Code / Codex / Gemini CLI / OpenHands / Takt など。ルールは `AGENTS.md` を共通正本とし、ツール固有設定は薄い委譲層です（[解説](concepts/multi-agent.md)）。

**Q15. 複数の AI を混在させても破綻しませんか？**  
しません。全員が `AGENTS.md` と `constitution.md` を共有正本にし、引き継ぎは会話履歴ではなく成果物で行います。

**Q16. AI に人間のアカウントで操作させてよいですか？**  
いけません。AI は専用のマシンアカウント（`@bot/*`）で行為します。これで「作成者≠承認者」を機械的に強制できます。

**Q17. AI が間違えたらどうなりますか？**  
AI の出力は**下書き**であり、人間レビューと品質ゲートを通って初めて反映されます。重大変更は人間承認が必須です。

**Q18. AI 全自動で開発できますか？**  
できません。要所で人間が判断する HITL（Human-in-the-Loop）が前提です。AI は起案・準備を自律し、反映は承認に従います。

**Q19. AI が生成したと、後から分かりますか？**  
分かります。コミットトレーラ（`Assisted-by:`）や PR ラベル `ai-generated`、ADR の起案者欄で識別します（監査証跡）。

## 仕様（spec）と SDD

**Q20. spec・plan・ADR の違いは？**  
spec = 何を・なぜ（What/Why）、plan = どう作るか（How）、ADR = なぜこの設計か（Why-this-design）。役割を分け重複させません（[解説](concepts/spec-driven-development.md)）。

**Q21. なぜコードより先に仕様を書くのですか？**  
仕様が「受け入れ基準」になり、実装の正しさを判定できるからです。実装ファーストは意図の喪失と手戻りを生みます。

**Q22. 仕様はどこに置きますか？**  
`specs/<NNN-feature-name>/spec.md`。3 桁連番 + kebab-case のディレクトリです。

**Q23. 仕様の粒度は？**  
1 機能（ユーザーに価値を届ける最小単位）= 1 spec ディレクトリ。横断的関心事は ADR か standards へ。

**Q24. 仕様と実装がズレたら？**  
実装を直すか仕様を更新するかの**どちらか**で解消します。どちらにするかは人間が判断します。

**Q25. `[NEEDS CLARIFICATION]` とは？**  
spec の未確定点に置くマーカーです。`/speckit.clarify` でつぶします。

**Q26. spec に設計を書いてよいですか？**  
いけません。設計（How）は plan、設計理由は ADR へ。spec は What/Why に徹します。

## ADR（設計判断の記録）

**Q27. ADR とは何ですか？**  
「なぜこの設計にしたか」を 1 ファイルに残す記録です（[解説](concepts/adr.md)）。採用案だけでなく却下案と理由も残します。

**Q28. ADR はいつ書きますか？**  
Class A / B（統治・セキュリティ・アーキテクチャ・公開API）や、長期的影響のある設計判断のときです。迷ったら書いてよい。

**Q29. ADR を書かなかったらどうなりますか？**  
重大変更（Class A/B）の PR には ADR 参照か「ADR 不要理由」が必要で、CI が記載の有無を検査します。書き忘れを検知できます。

**Q30. ADR のファイル名のルールは？**  
`adr-NNNN-short-title.md`（4 桁ゼロ埋め連番、小文字・ハイフン）。`0000` は予約。番号は再利用しません。

**Q31. ADR の Status にはどんな値が？**  
Proposed / Accepted / Rejected / Deprecated / Superseded。遷移は決められたものだけ許容されます。

**Q32. Accepted の ADR を後で書き換えてよいですか？**  
本文の実質的変更は不可。新しい ADR を起案して古いものを Superseded にします（誤記修正など軽微なものは変更履歴に記録の上で可）。

**Q33. 誰が ADR を承認しますか？**  
人間です。Accepted 化は Class A で、AI は起案（Proposed）まで。

**Q34. minimal と full、どちらを使う？**  
小規模・プロジェクト粒度なら minimal、組織・Class A 相当・監査要件が強い・定量比較が要るなら full。

**Q35. `adr/INDEX.md` は手で書きますか？**  
書きません。フロントマターから自動生成されます（`python scripts/generate_adr_index.py`）。CI が差分ゼロを検査します。

## Constitution とガバナンス

**Q36. Constitution（憲章）とは？**  
人間も AI も従う最上位ルールです（[解説](concepts/constitution.md)）。原則を定め、詳細は下位文書へ委譲します。

**Q37. 変更クラス A/B/C/D とは？**  
変更の重さの 4 分類です。AI の自律範囲・承認・ADR 要否・ゲートを決めます（[解説](concepts/governance.md)）。

**Q38. 自分の変更のクラスはどう決めますか？**  
まず対象パスで一次判定し、複数該当なら最も厳しいクラスを採用。内容（公開API・スキーマ・認可・外部依存・不可逆操作）でも引き上げます。

**Q39. クラスが分からないときは？**  
**Class A として扱います**（フェイルセーフ）。確定するまで最も厳格に扱うのが既定動作です。

**Q40. RFC 2119 のキーワードとは？**  
MUST / SHOULD / MAY など、義務の強さを表す語です。大文字のときだけ規範的意味を持ちます。

**Q41. 憲章は誰が変えられますか？**  
ガバナンス決定として、定められた承認者・定足数で改正します。AI は単独で承認・反映できません。

**Q42. Lite / Standard / Regulated の違いは？**  
統治の重さのプロファイルです。Lite=個人小規模、Standard=通常チーム（既定）、Regulated=規制・監査対象（[詳説](governance/index.md)）。

**Q43. プロファイルで何でも緩められますか？**  
いいえ。緩められるのは人間プロセスの重さだけ。絶対ルール（本番データ・作成者≠承認者・ゲート未通過マージ禁止・クラス未確定は A）は不変です。

**Q44. 「作成者≠承認者」とは？**  
変更の作成者と承認者を分ける職務分掌です。AI は自分が関わった変更を承認・自己マージできません。

## 品質ゲートと CI

**Q45. 品質ゲートとは？**  
ルールを CI で機械強制する仕組みです。開発者・AI・CI が同じ `task verify` を実行します（[解説](concepts/quality-gates.md)）。

**Q46. `task verify` と `task verify:fast` の違いは？**  
fast はコミット前の高速チェック、verify は push/PR/CI 向けの包括チェック（リンク・秘密情報・脆弱性・build を追加）。

**Q47. ローカルで通ったのに CI で落ちるのはなぜ？**  
本来は起きにくい設計です（同じコマンドを実行）。多くはツール未導入や環境差。[トラブルシューティング](troubleshooting.md) を参照。

**Q48. 失敗したゲートを一時的に無効化してよいですか？**  
回避目的では不可（MUST NOT）。原因側を直します。ゲート定義の変更は Class A で人間承認が必須です。

**Q49. コードがまだ無いのに CI が落ちませんか？**  
落ちません。コードが無くてもグリーンで通り、コードを足すと build/test/型/カバレッジが自動で有効化されます。

**Q50. カバレッジ基準はどこで決めますか？**  
`standards/testing-standards.md`。プロファイルにより初期値を調整できます。

## 導入・運用・ツール

**Q51. spec-kit は必須ですか？**  
必須ではありません。成果物はただの Markdown です。spec-kit があると作成フローを `/speckit.*` で回せます（[解説](concepts/spec-kit.md)）。

**Q52. spec-kit はどう入れますか？**  
公式 <https://github.com/github/spec-kit> を参照してください。位置づけは [前提環境](getting-started/prerequisites.md)。

**Q53. Windows でも使えますか？**  
使えます。品質ゲートのスクリプトは bash なので、**WSL2** か **Git Bash** を使ってください。

**Q54. `specs/001`・`002` は本物の機能ですか？**  
いいえ。GDPR を題材にした**サンプル**です。手本として残し、実プロジェクトでは置き換えます（[実例で学ぶ](examples/index.md)）。

**Q55. ブランチ保護は何を設定しますか？**  
`main` / `release/*` に、作成者以外の承認・include administrators・force-push 禁止・必須チェック `verify` を設定します（`ADOPTION.md`）。

**Q56. このドキュメントサイトはどう動かしますか？**  
`pip install -r requirements-docs.txt` の後 `mkdocs serve`。公開は GitHub Actions（[このサイトについて](about/site-design.md)）。
