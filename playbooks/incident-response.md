# Playbook: インシデント対応

* 上位規範: constitution.md（「7.」緊急時例外／「変更の可逆性とインシデント対応」）／ development-process.md「7.」
* 変更クラス: 本 Playbook 文書は Class C。**本番への緊急変更は Class A**（緊急承認者の承認必須）。

> 雛形です。採用組織が連絡体制・SLA・ツールに合わせて具体化してください。

## 前提

- 緊急承認者（incident commander）が指名されている（development-process.md「7.」）。
- ロールバック手段が用意されている。

## 検知・初動

1. 重大度を判定し、影響範囲・開始時刻を記録する。
2. 緊急承認者へエスカレーションする。AI は人間承認なしに本番変更してはならない（MUST NOT。緊急時も免除されない）。

## 収束

1. 暫定対応（ロールバック／フィーチャーフラグ無効化等）で影響を止める。
2. Break-glass を適用する場合、適用の事実・理由・範囲・承認者を記録する（constitution.md「7.」）。
3. 恒久対応を通常の変更フロー（クラス判定・レビュー）で実施する。

## 確認

- [ ] 観測性（ログ・メトリクス・トレース）で復旧を確認
- [ ] 本番 PII を AI／外部 AI に渡していない（MUST NOT）

## 事後

- **72 時間以内**に事後レビュー（スキップしたゲートの事後実行を含む）を完了する（constitution.md「7.」）。
- 原因・再発防止策を記録し、必要なら ADR またはガバナンス決定として残す。
